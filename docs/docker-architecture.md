# Docker Architecture (Compose, Current)

Last updated: 2026-03-10

## 1) Services in `docker-compose.yml`

1. `postgres`
   - image: `postgres:16-alpine`
   - exposed: `5432:5432`
   - healthcheck: `pg_isready`
2. `redis`
   - image: `redis:7-alpine`
   - exposed: `6379:6379`
   - healthcheck: `redis-cli ping`
3. `api`
   - build context: `./backend`
   - command: `npx prisma db push && node dist/main.js`
   - exposed: `4000:4000`
   - depends on healthy `postgres` + `redis`
4. `worker`
   - build context: `./backend`
   - command: `npx prisma db push && node dist/worker.js`
   - depends on healthy `postgres` + `redis`
   - current behavior is placeholder heartbeat loop, not queue processing
5. `frontend`
   - build context: `./frontend`
   - build arg: `VITE_GRAPHQL_URL`
   - exposed: `5173:80`
   - depends on `api`

## 2) Volumes and networking

- Named volume:
  - `pg_data` for Postgres persistence.
- Redis currently runs without a named volume in compose.
- Default compose network is used for service-to-service DNS (`postgres`, `redis`, `api`).

## 4) Startup sequence

1. `postgres` + `redis` start first.
2. `api` and `worker` wait for their healthchecks.
3. Both `api` and `worker` run `prisma db push` on startup.
4. `frontend` starts after `api` service container is up.

## 5) Runtime behavior in containers

1. User opens frontend container on port `5173`.
2. Frontend calls GraphQL at backend (`/graphql`) and REST routes (`/speech/analyze`, `/audio/:fileName`, `/health`).
3. API persists sessions/items/answers/reports/events in Postgres via Prisma.
4. Listening MP3 files are generated by API and served through `/audio/:fileName`.
5. Worker container runs but does not yet process Redis jobs.

## 6) Operational notes

1. `docker compose up --build` is the canonical local startup path.
2. Because `db push` runs on every API/worker startup, schema changes are applied automatically in dev.
3. Keep `.env` out of version control and never bake provider keys into images.
